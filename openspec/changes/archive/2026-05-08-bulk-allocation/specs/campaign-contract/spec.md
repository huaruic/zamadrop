# campaign-contract (delta) — bulk-allocation

## ADDED Requirements

### Requirement: Batched allocation primitive

The contract SHALL provide a `setAllocationsBatch(address[] recipients, externalEuint64[] encAmounts, bytes inputProof)` function that the admin uses to assign encrypted allocations to multiple recipients in a single transaction.

The function SHALL accept up to 16 recipients per call. The binding ceiling is the FHEVM HCU per-tx budget (the loop body's `FHE.add(_runningTotal, amount)` chain consumes Homomorphic Computation Unit depth tracked by `HCULimit.sol`); empirically, 32 recipients revert `HCUTransactionDepthLimitExceeded()` while 16 succeed. Higher counts SHALL fail at HCU enforcement before either the relayer-SDK packing limit (32 uint64) or the EVM block-gas limit (~50+ recipients) becomes relevant. The contract MAY safely accept arbitrarily large arrays; the limit is enforced upstream by client tooling.

For each recipient in the array, the function SHALL apply the same per-recipient semantics as the single-call `setAllocation`:
- Verify the recipient has not already received an allocation (`allocationSet[recipient] == false`)
- Verify and decode the encrypted amount via `FHE.fromExternal(encAmount, inputProof)` — the same `inputProof` is reused for every entry in the batch (the proof was generated to cover the whole packed input)
- Store the resulting ciphertext under `_allocation[recipient]`
- Grant FHE access to `address(this)` and `recipient`
- Add the amount to `_runningTotal`
- Set `allocationSet[recipient] = true`
- Increment `allocationCount` by 1
- Emit `AllocationSet(recipient)`

The function SHALL be admin-only and only callable in `State.Setup`. Any guard violation SHALL revert the entire transaction (atomic semantics — no partial state).

#### Scenario: Successful batch of 16 distinct recipients

- **GIVEN** state is `Setup`, no recipients have allocations yet, and the input ciphertext was generated by `relayer-sdk.createEncryptedInput().add64(amounts[0..15]).encrypt()`
- **WHEN** admin calls `setAllocationsBatch(recipients, encAmounts, inputProof)` with 16 entries
- **THEN** state remains `Setup`
- **AND** `allocationCount` increases by exactly 16
- **AND** `allocationSet[r] == true` for every `r` in `recipients`
- **AND** event `AllocationSet(r)` SHALL be emitted exactly once per recipient

#### Scenario: Duplicate within batch reverts atomically

- **GIVEN** state is `Setup`
- **WHEN** admin calls `setAllocationsBatch` with `recipients = [r1, r2, r1]` (third entry duplicates the first)
- **THEN** the call SHALL revert with `AllocationAlreadySet()`
- **AND** no state changes from this call SHALL persist (`allocationCount` unchanged, no `AllocationSet` events emitted)

#### Scenario: Recipient previously allocated reverts

- **GIVEN** `allocationSet[r1] == true` from a prior call
- **WHEN** admin calls `setAllocationsBatch(recipients, encAmounts, inputProof)` where `r1` appears in `recipients`
- **THEN** the call SHALL revert with `AllocationAlreadySet()`

#### Scenario: Mismatched array lengths revert

- **WHEN** `recipients.length != encAmounts.length`
- **THEN** the call SHALL revert with `ArrayLengthMismatch()`
- **AND** the revert SHALL happen before any per-recipient processing, regardless of whether `inputProof` would have been valid

#### Scenario: Non-admin call reverts

- **WHEN** any address other than `admin` calls `setAllocationsBatch`
- **THEN** the call SHALL revert with `NotAdmin()`

#### Scenario: Wrong state reverts

- **WHEN** `setAllocationsBatch` is called and `state != State.Setup`
- **THEN** the call SHALL revert with `NotSetup()`
- **AND** the revert applies regardless of which non-Setup state (Finalizing, Claiming, Failed, TimedOut after v8-finalize-recovery lands)

### Requirement: Batch invariants identical to single allocation

The contract SHALL guarantee that any sequence of `setAllocationsBatch` calls produces state identical to the equivalent sequence of single-call `setAllocation` calls, with the exception that gas usage and event ordering reflect the batched call structure.

This means:
- `allocationCount` after all batch calls equals the total number of distinct recipients allocated
- `_runningTotal` after all batch calls is FHE-equal to the sum of all amounts
- `_allocation[r]` for every allocated recipient holds the encrypted amount that was passed in
- `finalize()` accepts the post-batch state under the same conditions as the post-single-call state

#### Scenario: Batch and single-call paths converge

- **GIVEN** two campaigns deployed with identical constructor args
- **WHEN** campaign A receives 16 recipients via one `setAllocationsBatch` call
- **AND** campaign B receives the same 16 recipients via 16 sequential `setAllocation` calls
- **THEN** `campaignA.allocationCount() == campaignB.allocationCount() == 16`
- **AND** `campaignA.allocationSet(r) == campaignB.allocationSet(r) == true` for every recipient
- **AND** both campaigns SHALL accept `finalize()` under the same precondition (escrow funded ≥ declaredTotal)

### Requirement: Single-call setAllocation remains supported

The `setAllocation(address, externalEuint64, bytes)` function defined by V7 SHALL continue to function unchanged. Admin tooling that calls one allocation at a time (manual fixes, Foundry scripts, future migration tooling) SHALL not be broken by the addition of the batch variant.

#### Scenario: Mixed batch and single calls in same campaign

- **GIVEN** state is `Setup`
- **WHEN** admin calls `setAllocationsBatch` with 14 recipients
- **AND** then calls `setAllocation` with the 15th recipient (single)
- **AND** then calls `setAllocationsBatch` with 1 more recipient (a batch of one)
- **THEN** `allocationCount == 16`
- **AND** `finalize()` SHALL succeed if `recipientCount == 16` and escrow is funded
